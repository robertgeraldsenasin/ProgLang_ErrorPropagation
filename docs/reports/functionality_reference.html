Repository Functionality, Scripts, and Correctness Pipeline

Complete reference for how the package generates prompts, checks SQL, syncs logs, and produces analysis.

Generated package: final free-model Spider2-Lite SQLite error-propagation study repo

Repository root: ProgLang_ErrorPropagation

1. Architecture overview

The repository implements a manual version of the SELF-DEBUGGING pattern for Text-to-SQL: a prompt is generated from a real Spider2-Lite task, a human pastes the browser model’s reply, the code extracts SQL, executes it on SQLite, compares the returned rows to the oracle, labels the turn state, and builds the next feedback payload if needed.

5. Prompt creation and support-document handling

runner.py loads the selected task from Spider2-Lite, resolves the SQLite database, and asks schema_utils.py to dump the schema. It then asks prompt_context.py to resolve any external_knowledge files referenced by the task and injects those documents into the prompt. Turn 1 uses prompts/turn1_sql.txt; revision turns use one of the revise_F* templates. All revision prompts are self-contained, carrying the schema, support documents, question, previous SQL, and feedback payload in the same prompt.

Workbook Prompt Library: the tracked prompts are versioned and seeded from the actual prompt files.

6. Response parsing, SQL extraction, and execution

The operator can either paste the model reply directly and end with END, or enter FILE:/path/to/reply.txt. sql_extract.py looks first for a fenced sql block, then any fenced code block, then raw SQL. If none is found, the turn becomes FormatError.

executor.py then runs the extracted SQL in a child process with a fixed timeout. This isolates runaway queries and captures either rows/columns or an engine error string. The executor never guesses correctness from natural-language text.

7. Oracle loading and correctness determination

oracle.py first looks for an official gold execution-result file under spider2-lite/evaluation_suite/gold/exec_result. If none exists, it falls back to the official gold SQL under gold/sql and executes that locally. Predicted rows and oracle rows are normalized, and if neither side explicitly requests ordering, they are compared as sorted multisets. Matching rows produce Pass; successful execution with differing rows produces WrongResult; SQLite engine failures are classified into SyntaxError, SchemaError, or RuntimeError; timeout produces Timeout.

8. Feedback conditions

ProtocolFeedback contentUse in the package
F0PASS/FAIL onlyOptional baseline, not the headline configuration
F1SQLite engine error when execution fails; FAIL otherwisePilot pack
F2Engine errors or output-difference summary with row counts and sample rowsMain and holdout packs
F3Same as F2 plus explicit self-diagnosis instruction in the promptAvailable ablation

The package defaults to F1 for the pilot because it quickly exposes obvious syntax/schema issues, and to F2 for the main study because F2 is richer for wrong-result analysis.

9. Logging, workbook sync, and analysis exports

03_run_trajectory_manual.py writes machine logs to JSONL. 04_sync_logs_to_workbook.py upserts those JSONL rows into Run Plan, Turn Log, and Stability Checks, then rebuilds Run Summary and Dashboard. 05_generate_analysis_tables.py exports the turn-level, transition-level, run-level, model-level, protocol-level, and stability-level CSVs as well as the two primary PNG charts.

Main exports from script 05: turn_log.csv transition_counts.csv transition_matrix.csv state_distribution_by_turn.csv state_distribution_by_turn.png run_summary_from_logs.csv model_summary.csv protocol_summary.csv trajectory_summary.csv pass_rate_by_model.png run_plan.csv run_plan_raw.csv stability_checks.csv stability_summary.csv

Run pytest -q first. Then use the sample fixture to smoke-test the full operator workflow. Only after that should you validate the real Spider2 layout and begin pilot runs on the external dataset.

11. Proper operator workflow

  • Use script 07 on a new Windows device to recreate the environment and local dataset.
  • Run script 10 to reset and seed the workbook.
  • Run one trajectory with script 03.
  • If the run passed, run script 06; otherwise skip stability.
  • Run script 04 to sync the workbook and script 05 to regenerate analysis outputs.
  • Use script 08 if you want a single command that copies evidence and optionally commits changes.
