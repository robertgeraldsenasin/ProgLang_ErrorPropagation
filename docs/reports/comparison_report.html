1. Executive summary

This repository is finalized as a manual browser-based, zero-budget Text-to-SQL error-propagation study on the Spider2-Lite SQLite subset. The configured headline study uses four free-access browser models and a seeded workbook with 48 planned pilot-plus-main trajectories.

This is a manual browser benchmark, not a pure API benchmark. The package therefore logs the exact visible model label shown in each provider UI, preserves prompts and SQL artifacts per turn, and treats silent browser fallback as a study event rather than hiding it.

2. Why these free models fit this study

The study asks how models behave across a generate–execute–revise loop, not just whether they solve a task in one shot. That makes four properties more important than raw benchmark hype: strong multi-step revision ability, enough context to hold schema and support documents, easy free access for repeated manual tests, and visible model labels that can be logged honestly in the workbook.

4. Recommended run counts, turn budget, and stability budget

Final package protocol from configs/study_protocol_zero_budget.yaml.
ItemConfigured valueWhy this is the package default
Pilot tasks4Enough to debug the workflow before reporting headline numbers
Main tasks8Large enough to compare four models while staying realistic under free-tier limits
Pilot + main runs484 models × (4 pilot + 8 main) tasks
Optional holdout tasks4Used only after the main comparison is stable
Turn cap (T_max)4One initial attempt plus up to 3 repair turns
Stability checks per pass3 re-executionsEnough to detect fragile passes without excessive operator time

The turn cap is intentionally lower than the 10-turn maximum used in SELF-DEBUGGING because the paper also states that successful debugging mostly ended within 3 turns. In a zero-budget browser setup, a 4-turn cap preserves the self-debugging logic while staying realistic under free usage constraints—especially the ChatGPT Free GPT-5.3 limit.

6. Final operating process

  • Run the Windows bootstrap script on a fresh device to create .venv, clone Spider2 locally, download local_sqlite.zip, remove the nested Spider2/.git directory, validate the dataset, repair the workbook, and seed all reference sheets.
  • Use the 4-task pilot under protocol F1 to verify prompt quality, SQL extraction, workbook sync, and stability-check behavior.
  • Freeze the prompts and workbook once the pilot is behaving correctly.
  • Run the 8-task main comparison under protocol F2 across all four models.
  • For every passing run, execute 3 re-execution stability checks and sync the workbook again.
  • Export analysis CSVs and charts after each run block and copy selected evidence into docs/testing-evidence before committing.
  • Use the optional holdout tasks only after the main comparison is finished and stable.
Figure: the generate–execute–revise loop used in this repository is inspired by SELF-DEBUGGING, but adapted to Text-to-SQL and to a manual browser-based workflow.

7. Finished-paper positioning

With the package in its final state, the finished paper can honestly claim: (i) a reproducible manual browser benchmark on Spider2-Lite SQLite, (ii) a curated zero-budget task pack, (iii) turn-level logging of error propagation, regressions, and stability, and (iv) a workbook-plus-analysis workflow that maps every claim back to saved prompts, responses, SQL files, and JSONL logs.
